# nexus-booking CI/CD Pipeline

Documentation for the GitHub Actions CI/CD setup.

---

## Pipeline Overview

```
push/PR → lint → unit-tests → bdd-tests → property-tests
                                         → regression-tests
                                         → e2e-tests
                                         → docker-build
                           (main only) → deploy (GHCR push)
```

All jobs run in parallel where dependencies allow, minimising total
pipeline duration to ~4–5 minutes.

---

## Jobs

### `lint` — ruff code quality check

Runs `ruff check app/ tests/` with E, W, F (pyflakes), and I (imports).
Line-length violations (E501) are excluded — enforced by convention, not CI.

```yaml
- run: ruff check app/ tests/ --select E,W,F,I --ignore E501,W503
```

### `unit-tests` — pure unit tests

- **Marker**: `-m unit`
- **DB**: `sqlite+aiosqlite:///:memory:` (no external deps)
- **Coverage**: uploads `coverage-unit.xml` artifact
- **What it tests**: availability index logic, auth token helpers, Pydantic
  model validation, email HTML generation

### `bdd-tests` — Gherkin BDD scenarios

- **Marker**: `-m bdd`
- **DB**: SQLite in-memory
- **Feature files**: `tests/bdd/features/*.feature`
- **What it tests**: user-visible booking behaviours in natural language:
  booking a slot, double-booking rejection, admin cancellation, etc.

### `property-tests` — Hypothesis / DDT

- **Marker**: `-m property`
- **DB**: SQLite in-memory
- **What it tests**: availability index invariants and Pydantic validation
  hold for arbitrary inputs generated by Hypothesis, not just hand-picked
  examples.

### `regression-tests` — API contract stability

- **Marker**: `-m regression`
- **DB**: SQLite in-memory
- **What it tests**: response *shape* of all endpoints — field names, types,
  HTTP status codes. If any of these change, this job catches it before
  merge to prevent breaking the portfolio gateway integration.

### `e2e-tests` — end-to-end lifecycle

- **Marker**: `-m e2e`
- **DB**: SQLite in-memory
- **What it tests**: full booking journeys — create, retrieve, cancel, verify
  slot freed, delete, concurrent bookings.

### `docker-build` — multi-stage Docker build

- Runs after `unit-tests` and `regression-tests` pass
- Uses `docker/build-push-action` with GitHub Actions cache (`type=gha`)
- Does **not** push on PRs — build-only validation

### `deploy` — push to GHCR (main only)

Triggered only on push to `main`. Pushes two tags:
- `ghcr.io/itkdaniel/nexus-booking:latest`
- `ghcr.io/itkdaniel/nexus-booking:<git-sha>`

Requires the `production` GitHub Environment to be configured.

## Environment Variables (CI)

All test jobs run with:

```yaml
env:
  DATABASE_URL: "sqlite+aiosqlite:///:memory:"
  JWT_SECRET: "ci-test-secret"
  EMAIL_ENABLED: "false"
```

No real database or SMTP server is required in CI.

---

## Caching

- **pip**: `actions/setup-python@v5` with `cache: pip` — caches
  `~/.cache/pip` keyed on `requirements*.txt`
- **Docker layers**: `type=gha` GitHub Actions cache — dramatically reduces
  Docker build time on repeated runs

## Secrets Required

| Secret | Used by | Description |
|--------|---------|-------------|
| `GITHUB_TOKEN` | `deploy` | Auto-provided by GitHub Actions |

For production deployments, add to the `production` environment:

| Secret | Description |
|--------|-------------|
| `DATABASE_URL` | Production PostgreSQL connection string |
| `JWT_SECRET` | Production HMAC signing key |

## Adding a New CI Job

1. Edit `.github/workflows/ci.yml`
2. Add the job under `jobs:`
3. Use `needs: [unit-tests]` if the job depends on unit tests passing
4. Use `if: github.ref == 'refs/heads/main'` for main-only jobs

```yaml
my-new-job:
  name: My New Job
  runs-on: ubuntu-latest
  needs: [unit-tests]
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with:
        python-version: "3.12"
        cache: pip
    - run: pip install -r requirements-dev.txt
    - run: pytest tests/my_new_suite/ -m my_marker
      env:
        DATABASE_URL: "sqlite+aiosqlite:///:memory:"
        JWT_SECRET: "ci-test-secret"
        EMAIL_ENABLED: "false"
```

## Local CI Simulation

```bash
# Lint
ruff check app/ tests/

# All test categories
pytest tests/unit/       -m unit       -v
pytest tests/bdd/        -m bdd        -v
pytest tests/property/   -m property   -v
pytest tests/regression/ -m regression -v
pytest tests/e2e/        -m e2e        -v

# Full run (mirrors CI)
pytest -v

# Docker build
docker build -t nexus-booking:local .
```
